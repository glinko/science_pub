# Спецификация: live-seed кнопка в review dashboard

## Контекст

Проект `science-pub` уже умеет:

- собирать статьи из arXiv через job `collect-arxiv`;
- присваивать статьям score через job `score-papers`;
- показывать статьи в review dashboard;
- давать пользователю ручной workflow `approve / reject`.

Сейчас в dashboard уже можно работать с тем, что есть в базе, но не хватает простого входа для наполнения системы свежими статьями прямо из UI. Пользователь уже подтвердил, что ручная работа с dashboard удобна, и следующий шаг должен убрать зависимость от отдельного ручного запуска backend jobs.

## Цель

Добавить в review dashboard одну кнопку, которая:

- запускает сбор свежих статей из arXiv;
- после успешного завершения автоматически запускает scoring;
- показывает пользователю прогресс цепочки;
- автоматически обновляет список статей после завершения;
- не требует нового orchestration endpoint на backend.

Результат должен быть пригоден для ежедневного сценария: открыть dashboard, нажать одну кнопку, дождаться завершения, перейти к review свежих статей.

## Зафиксированные решения

### 1. Кнопка одна, а не две

В dashboard добавляется одна кнопка верхнего уровня, например `Fetch Fresh Papers`.

Причина:

- пользователю не нужен технический split между collect и score;
- цель этой функции не дать контроль над внутренней очередью, а быстро наполнить review-поверхность;
- две отдельные кнопки добавили бы лишний ручной шаг и ухудшили UX.

### 2. Оркестрация живет во frontend

Новый backend endpoint для orchestration не добавляется.

Dashboard переиспользует уже существующие endpoints:

- `POST /jobs/collect-arxiv`
- `POST /jobs/score-papers`
- `GET /jobs`

Выбранный подход:

- frontend создает collect job;
- frontend poll'ит список jobs;
- после `succeeded` у collect job frontend создает score job;
- после `succeeded` у score job frontend обновляет список статей.

Причина:

- это самый короткий путь к рабочему UX;
- backend уже содержит все необходимые job primitives;
- не появляется новый orchestration слой, который пока не нужен.

### 3. Кнопка использует defaults milestone 1

По умолчанию collect запускается без пользовательской формы параметров:

- `categories = []`, что означает backend default categories;
- `max_results = 100`.

Score запускается так:

- `limit = 20`;
- `status = "collected"`;
- `provider = "mock"`.

Причина:

- пользователь просил именно seed-entrypoint, а не полноценный control panel для pipeline;
- сложная настройка категорий и provider'ов не нужна для первого UX-шага;
- если в будущем понадобится ручная настройка, ее можно добавить отдельной итерацией.

### 4. UI обязан показывать этап выполнения

Пользователь должен видеть не просто spinner, а понятное состояние цепочки.

Минимальный набор состояний:

- `idle`
- `collecting`
- `scoring`
- `success`
- `failed`

Для `failed` UI должен показывать понятное сообщение об ошибке, если backend job содержит `error_text`.

### 5. Во время цепочки повторный запуск запрещен

Пока dashboard выполняет live-seed цепочку, кнопка disabled.

Причина:

- это предотвращает случайный запуск нескольких collect/score проходов подряд из одного UI;
- этого достаточно для первой версии;
- сложная дедупликация параллельных запусков в UI сейчас не нужна.

### 6. После успеха UI обновляет статьи автоматически

После успешного завершения score job dashboard:

- повторно загружает `GET /papers` с текущими фильтрами;
- сохраняет текущее состояние фильтров;
- не сбрасывает пользователя в "чистый" экран;
- по возможности сохраняет выбранную статью, если она все еще присутствует в выдаче.

Цель:

- новый контент появляется без ручного refresh;
- пользователь не теряет контекст review.

## Подробное поведение

### Пользовательский сценарий

1. Пользователь открывает dashboard.
2. Нажимает `Fetch Fresh Papers`.
3. UI сразу показывает состояние `Collecting...`.
4. Frontend вызывает `POST /jobs/collect-arxiv`.
5. Frontend начинает polling `GET /jobs`.
6. Когда collect job получает статус `succeeded`, frontend вызывает `POST /jobs/score-papers`.
7. UI переключается в состояние `Scoring...`.
8. Frontend продолжает polling `GET /jobs`.
9. Когда score job получает статус `succeeded`, UI:
   - показывает `Done`;
   - повторно загружает список статей;
   - сохраняет detail для текущей статьи, если она осталась в выдаче;
   - сбрасывает detail в нейтральное состояние, если текущая статья больше не проходит под активные фильтры.
10. Пользователь видит свежие статьи и начинает review.

### Поведение при ошибке

Если collect job завершился `failed`:

- scoring не запускается;
- UI показывает ошибку collect job;
- кнопка снова становится доступной.

Если score job завершился `failed`:

- UI показывает ошибку score job;
- список статей не считается автоматически обновленным;
- кнопка снова становится доступной.

Если polling временно не удался:

- UI показывает общий network/backend error;
- цепочка считается failed для текущей попытки;
- пользователь может попробовать снова.

## Архитектурные изменения

### Frontend

Нужны следующие изменения:

- новый API слой для jobs:
  - enqueue collect;
  - enqueue score;
  - list jobs;
- локальная orchestration-логика в `App.tsx` или рядом с ним;
- header action button;
- небольшой status banner или inline status block для live-seed состояния.

Frontend не хранит отдельный persistent state для jobs. Все основано на runtime state и данных из `GET /jobs`.

### Backend

Новый endpoint не требуется.

Backend меняется только если выяснится, что текущих response fields недостаточно для frontend orchestration. На данный момент ожидается, что существующий `JobResponse` уже достаточен:

- `id`
- `job_type`
- `status`
- `error_text`
- `created_at`
- `updated_at`

## Границы ответственности

### Что входит в эту итерацию

- кнопка live-seed в dashboard;
- frontend orchestration `collect -> score`;
- polling job statuses;
- статус выполнения в UI;
- auto-refresh списка статей после успеха;
- frontend tests на новую orchestration-логику;
- документация по новому workflow.

### Что сознательно не входит

- форма выбора arXiv categories;
- выбор `max_results` из UI;
- выбор scoring provider из UI;
- отдельный backend endpoint `refresh-papers`;
- realtime push через websockets или SSE;
- history panel всех seed attempts;
- защита от параллельных запусков между разными браузерами или пользователями;
- chaining следующих этапов pipeline после scoring.

Последний пункт важен: эта кнопка закрывает только seed вход в review dashboard, а не полный pipeline до `analyze/script/render`.

## Тестовая стратегия

### Frontend

Нужно покрыть минимум:

- кнопка рендерится в dashboard header;
- по нажатию отправляется collect job;
- после успешного collect job отправляется score job;
- во время выполнения кнопка disabled;
- при успехе список статей reload'ится;
- при failed collect score job не запускается;
- при failed score показывается error state.

### Smoke

Минимальный smoke после реализации:

- открыть dashboard;
- нажать кнопку live-seed;
- убедиться, что UI показывает `Collecting...`, затем `Scoring...`, затем success;
- убедиться, что `/jobs` отражает оба job;
- убедиться, что список статей обновился без ручного reload.

## Почему это правильный следующий шаг

Dashboard уже доказал полезность как review-поверхность. Следующая практическая проблема не в новом editorial UI, а в том, что пополнение статей пока зависит от внешнего ручного запуска.

Кнопка live-seed решает именно этот operational gap:

- делает dashboard самодостаточной рабочей точкой;
- усиливает ценность уже существующих collect/score jobs;
- не требует тяжелой backend orchestration;
- сохраняет scope небольшим и прикладным.

Это хороший мост между текущим review milestone и следующим этапом pipeline, где уже можно заниматься более глубокими post-score сценариями.
