# Спецификация: review-dashboard для статей в `science-pub`

## Контекст

Проект `science-pub` уже умеет:

- собирать статьи из arXiv;
- хранить их в `papers`;
- выставлять им score;
- фильтровать и получать статьи через `GET /papers`;
- менять статус статьи через `PATCH /papers/{id}/status`.

На текущем этапе в проекте не хватает рабочего интерфейса ручного review. Пользователю нужен дашборд, где можно:

- видеть все статьи;
- фильтровать их по ключевым признакам;
- открывать одну статью детально;
- утверждать или отклонять ее.

Это соответствует общей идее проекта: автоматический pipeline помогает отбирать материалы, но финальное editorial-решение остается ручным.

## Цель первой версии

Добавить отдельный frontend dashboard для review статей, который работает поверх существующего backend API и дает базовый editorial workflow:

- просмотр списка статей;
- фильтрация;
- detail view;
- действия `Approve` и `Reject`.

Первая версия должна быть маленькой, быстрой в реализации и не превращаться в полноценную editorial platform.

## Зафиксированные решения

### 1. Какие статьи видны в дашборде

В первой версии dashboard показывает все статьи, а не только отдельную review-очередь.

Основание:

- пользователь прямо хочет видеть все статьи;
- это упрощает навигацию по pipeline;
- необходимость в отдельной `selected-only` витрине пока отсутствует.

Управление потоком делается через фильтры, а не через несколько разных экранов.

### 2. Основной layout

Для первой версии выбран layout `Table + Detail`.

Это означает:

- слева или в основной колонке расположен список статей;
- справа находится detail-панель выбранной статьи;
- пользователь не уходит на отдельную detail-страницу для каждого review-действия.

Почему выбран именно этот вариант:

- он лучше всего балансирует массовый просмотр и внимательное чтение одной статьи;
- он естественно поддерживает фильтры;
- он хорошо подходит под быстрые approve/reject действия.

### 3. Decision model

В первой версии действия дашборда маппятся так:

- `Approve -> approved`
- `Reject -> rejected`

Комментарий к решению в этой версии не требуется.

Это решение сознательно упрощает UX:

- не нужны дополнительные формы;
- не нужны обязательные notes;
- не возникает лишней friction при ручном review.

### 4. Фильтры первой версии

В первой версии обязательны следующие фильтры:

- `status`
- `source`
- `category`
- `min_score`
- `search`

`search` должен работать по крайней мере по:

- `title`
- `source_id`

Остальные фильтры и сортировки из существующего API можно сохранить, но они не считаются ключевыми UX-элементами первой версии.

### 5. Архитектурный подход

Dashboard реализуется как отдельное маленькое frontend-приложение в репозитории, а не как server-rendered экран внутри backend.

Основание:

- frontend не смешивается с FastAPI runtime;
- UI можно развивать независимо;
- легче добавлять новые review-сценарии без захламления backend шаблонами;
- API уже в основном существует, значит frontend логично строить поверх него.

### 6. Источник истины

Backend остается единственным источником истины.

Frontend:

- не хранит бизнес-состояние статей вне runtime UI;
- не дублирует editorial-логику;
- использует backend API для списка, detail и смены статуса.

## Рекомендованная архитектура

### Frontend

В репозитории появляется отдельное frontend-приложение для dashboard.

Минимальный состав ответственности:

- экран списка статей;
- панель фильтров;
- detail-панель выбранной статьи;
- кнопки `Approve` / `Reject`;
- локальные loading/error/empty states.

Первая версия не требует сложного роутинга. Достаточно одного основного review-экрана.

### Backend

Существующий backend в целом переиспользуется, но для dashboard нужна как минимум одна точечная доработка.

### Уже подходящие endpoints

- `GET /papers`
- `GET /papers/{id}`
- `PATCH /papers/{id}/status`

### Обязательная доработка API

Нужен серверный `search`, чтобы dashboard мог искать статьи без клиентских обходных решений поверх пагинации.

`search` должен добавляться в `GET /papers` и фильтровать по:

- `title`
- `source_id`

При желании можно включить и `abstract`, но это не является обязательным требованием первой версии.

## Предлагаемый UI flow

### 1. Начальная загрузка

При открытии dashboard пользователь видит:

- список статей;
- фильтры;
- пустую или предварительно выбранную detail-панель.

Рекомендуемое поведение:

- таблица загружается сразу;
- detail-панель показывает первую статью из результата или нейтральное состояние "выберите статью".

### 2. Выбор статьи

Клик по строке статьи:

- выделяет статью в списке;
- открывает ее полные данные в detail-панели;
- не требует отдельной навигации на новый route в первой версии.

### 3. Approve / Reject

При нажатии:

- `Approve` отправляется `PATCH /papers/{id}/status` со статусом `approved`;
- `Reject` отправляется `PATCH /papers/{id}/status` со статусом `rejected`.

После успешного ответа:

- статус в detail-панели обновляется сразу;
- строка в таблице обновляется сразу;
- если статья больше не подходит под активный фильтр, она может исчезнуть из списка автоматически.

Это считается нормальным UX-поведением review-flow, а не ошибкой.

## Состояния интерфейса

Первая версия должна явно поддерживать:

- initial loading state;
- loading при смене фильтров;
- loading при approve/reject действии;
- empty state, если фильтры не вернули результатов;
- error state, если backend недоступен;
- disabled state у action-кнопок на время запроса.

Никаких модалок, forced confirmations и сложных optimistic rollback-сценариев в первой версии не требуется.

## Scope первой версии

### Что входит

- отдельный frontend dashboard;
- layout `Table + Detail`;
- список всех статей;
- фильтры `status`, `source`, `category`, `min_score`, `search`;
- detail-панель статьи;
- approve/reject действия;
- server-side `search` в backend;
- базовые loading / empty / error states;
- документация по новому review-слою.

### Что сознательно не входит

- комментарии к решениям;
- обязательные reject notes;
- bulk actions;
- drag-and-drop kanban board;
- auth и роли;
- audit trail решений;
- rich editorial notes;
- inline editing summary/script;
- realtime collaboration;
- live updates через websockets;
- персональные очереди по пользователям.

## Данные, которые важно показать в detail-панели

Для первой версии detail-панель должна показывать минимум:

- `title`
- `source`
- `source_id`
- `published_at`
- `status`
- `authors`
- `categories`
- `abstract`
- latest score, если он есть
- `model_used` для latest score, если он есть

Этого достаточно, чтобы пользователь мог принять простое editorial-решение без перехода в другие экраны.

## Данные, которые важно показать в списке

В таблице должны быть минимум:

- title
- status
- source
- category или основной category marker
- final score
- published date

Главный принцип: список должен быть пригоден для быстрого сканирования, а не для чтения всей статьи.

## Deployment-подход первой версии

Для production первой версии лучше держать frontend отдельным маленьким сервисом в compose, а не встраивать UI в backend runtime.

Это решение:

- не смешивает API и UI в одном процессе;
- делает rollout более предсказуемым;
- оставляет backend чистым;
- упрощает будущие фронтенд-итерации.

При этом приложение остается маленьким и не требует сложной platform-архитектуры.

## Тестовая стратегия

Нужны проверки минимум на следующие вещи:

### Backend

- `GET /papers` поддерживает `search`;
- поиск корректно фильтрует по `title` и `source_id`;
- `PATCH /papers/{id}/status` продолжает корректно переводить статью в `approved` и `rejected`.

### Frontend

- список статей загружается;
- фильтры меняют query;
- выбор строки открывает detail-панель;
- approve/reject обновляют статус;
- empty state и error state отображаются корректно.

## Почему это правильный следующий шаг

Сейчас проект хорошо умеет автоматизировать ранние части pipeline, но почти не дает человеку удобной рабочей поверхности для editorial decision-making.

Этот dashboard закрывает именно этот разрыв:

- не меняет core backend-модель проекта;
- не требует тяжелого product-scope;
- дает реальную ценность уже на первой версии;
- логично продолжает идею "automation + human review".

Именно поэтому review-dashboard должен идти раньше, чем сложная media-автоматизация, notes-system или полноценная moderation platform.
