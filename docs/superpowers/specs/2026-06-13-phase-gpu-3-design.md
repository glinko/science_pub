# Спецификация: Phase GPU-3 - интеграция основного стека с GPU-нодой

## Контекст

Проект `science-pub` уже развернут на основном узле `sci_docker` (`192.168.88.150`) и содержит рабочий стек Milestone 1: `postgres`, `redis`, `minio`, `qdrant`, `backend`, `worker`, `n8n`, `litellm`.

Дополнительно подготовлена GPU-нода `llm` (`192.168.88.20`) со следующими уже работающими сервисами:

- `qwen3.6` на `http://192.168.88.20:9000/v1`
- `bge-m3` на `http://192.168.88.20:9001/v1`
- `agentmemory-openai-mux` на `http://192.168.88.20:9100`

На GPU-ноде уже установлен Docker Engine и NVIDIA Container Toolkit, но в этой фазе существующие systemd-сервисы не переносятся в Docker.

## Цель фазы

Сделать основной стек `science-pub` готовым к использованию внешних GPU-сервисов через LiteLLM без миграции самих моделей и без расширения бизнес-логики за пределы readiness-интеграции.

Фаза считается успешной, если:

- основной стек знает о GPU-ноде через явный конфиг;
- LiteLLM маршрутизирует запросы к существующим upstream endpoint'ам GPU-ноды;
- backend умеет показывать состояние GPU-интеграции в health/config endpoints;
- `LiteLLMProvider` перестает быть заглушкой readiness-уровня и умеет выполнять реальный HTTP-вызов к LiteLLM;
- текущий milestone-поток не ломается, а scoring по умолчанию остается на `mock`.

## Зафиксированные решения

### 1. Переиспользование существующих моделей

Вместо разворачивания новых inference-сервисов используются уже работающие сервисы на GPU-ноде.

На этой фазе:

- `fast-small` временно указывает на `qwen3.6` на `:9000`;
- `deep-analysis` тоже временно указывает на `qwen3.6` на `:9000`;
- `embeddings` указывает на `bge-m3` на `:9001`.

Это временное решение допустимо, потому что цель фазы - не качество модельного роутинга, а рабочая интеграционная связка между основным стеком и GPU-нодой.

### 2. Тонкая интеграция без прокси-оберток

В этой фазе не вводятся дополнительные proxy/container wrapper сервисы на GPU-ноде. Основной стек подключается напрямую к уже работающим OpenAI-compatible endpoint'ам через LiteLLM.

### 3. Без ограничения доступа на GPU-ноде

Сетевой доступ к сервисам GPU-ноды в этой фазе дополнительно не ужесточается. Это зафиксированное пользовательское решение для локальной реализации.

### 4. Без миграции systemd-сервисов в Docker

`qwen3.6`, `bge-m3` и `agentmemory-openai-mux` продолжают работать как есть. `docker-compose.gpu.yml` и `.env.gpu.example` на GPU-ноде приводятся к согласованному виду, но не становятся источником истины для уже работающих inference-сервисов.

## Архитектура

### Основной узел `sci_docker`

На основном узле остаются:

- API backend
- worker
- Redis/RQ
- Postgres
- MinIO
- Qdrant
- LiteLLM

LiteLLM становится точкой маршрутизации между приложением и внешними GPU endpoint'ами.

### GPU-нода `llm`

GPU-нода предоставляет существующие upstream endpoint'ы:

- текстовая модель: `http://192.168.88.20:9000/v1`
- embeddings: `http://192.168.88.20:9001/v1`
- дополнительный mux-сервис: `http://192.168.88.20:9100`

Сервис на `:9100` документируется как доступный, но в обязательный поток этой фазы не включается.

### Поток вызова

1. Backend или worker вызывает LiteLLM во внутренней сети compose.
2. LiteLLM выбирает alias модели.
3. LiteLLM проксирует запрос на внешний endpoint GPU-ноды.
4. GPU upstream возвращает OpenAI-compatible ответ.
5. Backend либо использует результат, либо возвращает контролируемую ошибку провайдера.

## Изменения в конфигурации

### Корневой `.env.example`

В репозитории появляются явные переменные GPU-интеграции:

- `GPU_NODE_HOST`
- `GPU_LLM_FAST_URL`
- `GPU_LLM_DEEP_URL`
- `GPU_EMBEDDINGS_URL`
- `GPU_TTS_URL`
- `GPU_COMFYUI_URL`
- `GPU_RENDER_URL`

Значения по умолчанию в примере должны соответствовать локальной топологии:

- `GPU_NODE_HOST=192.168.88.20`
- `GPU_LLM_FAST_URL=http://192.168.88.20:9000/v1`
- `GPU_LLM_DEEP_URL=http://192.168.88.20:9000/v1`
- `GPU_EMBEDDINGS_URL=http://192.168.88.20:9001/v1`

`GPU_TTS_URL`, `GPU_COMFYUI_URL`, `GPU_RENDER_URL` могут быть заданы как readiness-ориентированные заготовки, даже если бизнес-поток пока их не использует.

### `docker-compose.yml`

LiteLLM получает конфиг, ориентированный на внешние upstream endpoint'ы GPU-ноды. Compose не должен предполагать, что inference-сервис живет в том же compose stack.

### `infra/litellm/config.yaml`

Добавляются стабильные alias-модели приложения:

- `gpu/fast-small`
- `gpu/deep-analysis`
- `gpu/embeddings`

Каждый alias должен однозначно маппиться на свой upstream URL и модельную роль. На этой фазе первые два alias допустимо указывают на один и тот же `qwen3.6` endpoint.

## Изменения в backend

### Settings и проверка конфигурации

Backend settings расширяются новыми GPU/LiteLLM параметрами. Конфиг должен валидироваться централизованно и явно сообщать:

- какие GPU endpoint'ы заданы;
- какие из них обязательны для текущей фазы;
- какие заданы только как будущая readiness-заготовка.

### `LiteLLMProvider`

`LiteLLMProvider` перестает быть отложенной заглушкой readiness-фазы и становится рабочим HTTP-клиентом до LiteLLM.

Требования:

- использовать OpenAI-compatible `/chat/completions` или эквивалентный LiteLLM endpoint;
- принимать `prompt` и `model`;
- возвращать строковый ответ;
- поднимать контролируемую доменную ошибку при сетевом сбое, невалидном ответе или отсутствии модели.

Бизнес-логика по умолчанию не переводится массово на этот provider. Основной scoring milestone 1 остается на `mock`, чтобы не ломать детерминированные проверки.

### Provider registry

Реестр провайдеров сохраняет минимум:

- `mock`
- `litellm`

`litellm` теперь считается готовым инфраструктурно и на уровне простого вызова, но не становится обязательным дефолтом для всех существующих фоновых сценариев.

## Health, readiness и деградация

### `GET /health`

Endpoint сохраняет `200` для общего живого состояния приложения, если локальные зависимости рабочего стека доступны.

При этом в деталях ответа должны быть отдельно отражены:

- состояние `postgres`
- состояние `redis`
- состояние `minio`
- состояние `qdrant`
- состояние `litellm`
- состояние внешнего GPU LLM upstream
- состояние внешнего GPU embeddings upstream

Если LiteLLM или GPU upstream недоступен, ответ не должен маскировать проблему, но и не должен валить весь backend-процесс.

### `GET /config-check`

Endpoint должен явно различать:

- конфиг валиден;
- LiteLLM доступен;
- GPU text upstream доступен;
- GPU embeddings upstream доступен;
- TTS/ComfyUI/Render заданы только как future-ready параметры;
- интеграция частично готова, если заготовленные URL существуют в конфиге, но не используются в текущем business flow.

### Ошибки провайдера

Сетевые и интеграционные ошибки должны возвращаться как контролируемые provider-level ошибки, пригодные для API-ответов и журналирования worker'ов. Недоступность GPU endpoint'а считается degraded-инцидентом, а не причиной падения всего сервиса.

## Изменения на GPU-ноде

На GPU-ноде меняются только инфраструктурные артефакты сопровождения:

- `/opt/science-video-factory/.env.gpu.example`
- `/opt/science-video-factory/docker-compose.gpu.yml`

Цель этих изменений:

- отразить фактические endpoint'ы;
- закрепить единый layout для следующих фаз;
- не вмешиваться в текущие systemd-сервисы.

Подъем новых GPU-контейнеров не является обязательным результатом этой фазы.

## Тестирование и верификация

### Локально в репозитории

Автотесты должны покрывать:

- валидацию новых settings;
- выбор provider через registry;
- поведение `LiteLLMProvider` на успешном и ошибочном HTTP-ответе;
- структуру `config-check` для readiness/degraded случаев.

### На основном узле `sci_docker`

Проверяются:

- `docker compose config`
- `docker compose up -d --build`
- здоровье LiteLLM-контейнера
- ответы backend endpoint'ов `/health`, `/version`, `/config-check`

### End-to-end

Проверяется успешный тестовый вызов по alias-модели через LiteLLM с реальным ответом от GPU-ноды:

- `gpu/fast-small`
- при наличии отдельного smoke-сценария также `gpu/deep-analysis`

Embeddings endpoint должен проходить как минимум readiness-проверку и при возможности минимальный smoke-test запроса.

## Что не входит в Phase GPU-3

Вне рамок этой фазы:

- выбор и загрузка отдельной маленькой модели вместо временного reuse `qwen3.6`;
- перенос inference-сервисов GPU-ноды в Docker Compose;
- обязательное включение `agentmemory-openai-mux` в production flow;
- реальная интеграция TTS, ComfyUI и render pipeline;
- переключение основного scoring с `mock` на реальную LLM-оценку;
- orchestration для тяжелых GPU jobs.

## Риски и ограничения

- `fast-small` и `deep-analysis` временно используют один и тот же upstream, поэтому логическое разделение ролей пока носит архитектурный, а не ресурсный характер.
- Текущие сервисы GPU-ноды работают вне Docker, поэтому их жизненный цикл управляется отдельно от compose-стека основного проекта.
- Внешний сетевой вызов между `192.168.88.150` и `192.168.88.20` добавляет новую точку отказа, которую нужно явно показывать в readiness.

## Критерий завершения

Фаза GPU-3 завершена, когда:

- все конфигурационные точки GPU-интеграции заведены в репозитории;
- LiteLLM на основном узле умеет стабильно проксировать вызовы на существующие GPU endpoint'ы;
- backend отражает readiness и degraded-состояния GPU-интеграции;
- `LiteLLMProvider` работает как реальный интеграционный слой;
- документация фазы и связанные GPU docs обновлены на русском языке;
- существующий milestone 1 не сломан и тесты проходят.
